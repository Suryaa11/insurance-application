INSURANCE MANAGEMENT APPLICATION - COMPLETE PROJECT OVERVIEW

1. PROJECT SUMMARY

This repository is a full-stack Insurance Management Application built with:
- Frontend: React, Vite, React Router, Axios, Material UI, Context API
- Backend: Node.js, Express.js, JWT authentication, Multer, bcryptjs, express-validator, Winston
- Database: MongoDB with Mongoose ODM
- Storage: Azure Blob Storage for document binaries, MongoDB for metadata
- DevOps: Docker, Docker Compose, environment variables, health checks

The application supports two roles:
- CUSTOMER
- ADMIN

The system covers the full insurance workflow:
- Customers register, login, browse plans, apply for insurance, upload documents, and track status
- Admins manage plans, review applications, verify documents, approve or reject applications, and view reports/dashboards

2. CORE BUSINESS GOALS

The app is designed to manage insurance onboarding and verification in one place:
- Create and maintain insurance plans
- Allow customers to submit applications against a selected plan
- Allow customers to upload required documents
- Let admins verify documents before approval
- Notify customers about key events
- Provide dashboards for both customer and admin users
- Support reporting, filtering, search, and pagination for admin workflows

5. BACKEND ARCHITECTURE

The backend follows a layered structure:
- Routes: define REST endpoints and middleware chains
- Controllers: receive requests and call services
- Services: contain business logic
- Repositories: encapsulate database access
- Models: Mongoose schemas
- Middleware: authentication, validation, uploads, and error handling
- Utils: reusable helpers

Main backend boot flow:
- backend/src/server.js starts the app
- backend/src/config/env.js loads and validates environment variables
- backend/src/config/db.js connects to MongoDB
- backend/src/app.js configures Express, security middleware, logging, static upload serving, routes, and error handlers

8. BACKEND FILE UPLOAD DESIGN

Uploads are stored in Azure Blob Storage.

The upload middleware:
- Uses multer memory storage
- Restricts file types to PDF, JPG, PNG
- Enforces a 10 MB file size limit
- Streams buffers to Azure Blob

Static serving:
- The backend generates short-lived signed Azure access URLs for viewing documents

9. BACKEND SECURITY

Implemented security controls:
- JWT auth for protected routes
- Refresh token support
- bcryptjs password hashing
- helmet for secure headers
- cors configured by CLIENT_ORIGIN
- express-rate-limit for request throttling
- express-validator for request validation
- RBAC for admin and customer access

10. BACKEND ERROR HANDLING

The backend returns consistent JSON responses:

  "success": true,

  "data": {}

Error handling includes:
- Custom ApiError class

- Validation middleware
- Global not-found middleware
- Global error handler
- Winston logging for request/error visibility

15. FRONTEND AUTH BEHAVIOR

Authentication flow:
- Login and register call backend auth endpoints
- accessToken, refreshToken, and user are stored in localStorage
- Axios request interceptor injects the access token
- Axios response interceptor uses refresh token flow on 401 responses
- If refresh fails, local auth state is cleared

Protected access:
- ProtectedRoute prevents unauthenticated access
- Role checks prevent customers from entering admin routes and vice versa

17. LOCAL DEVELOPMENT

Recommended local setup:
1. Install workspace dependencies:
   - npm install --workspaces
2. Seed the database:
   - npm run seed --workspace backend
3. Run the backend:
   - npm run dev --workspace backend
4. Run the frontend:
   - npm run dev --workspace frontend

Windows convenience scripts:
- start-local.ps1 starts MongoDB service, seeds data, and launches backend + frontend
- stop-local.ps1 stops the launched local processes

Important local assumption:
- The launcher expects a local MongoDB Windows service named MongoDB running on localhost:27017

22. CURRENT LIMITATIONS / ASSUMPTIONS

- The project is built without a test suite.
- MongoDB is assumed to be available locally as a Windows service for the start-local.ps1 flow.
- Uploaded files are stored in Azure Blob Storage while MongoDB keeps metadata only.
- The frontend bundle is split with lazy loading, but the app still uses a single Vite SPA deployment.
- Some business workflows can be expanded further, such as richer document rejection reasons, audit history, and email notifications.
